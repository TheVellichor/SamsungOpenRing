# Session Log — 2026-06-19

A deep-dive that started with "my ring's battery dies in a day, why?" and ended with
a full reverse-engineering of the ring's SmartThings BLE security stack. This log
records what was done, in order, and where each result lives.

Device: ring SM-Q509 (`D0:56:FB:71:B3:67`) ↔ phone SM-S928B (Android 16).

---

## 1. Battery diagnosis (from the existing project)

Re-read the prior project (PROJECT-LOG.md, RFC, the two `btsnoop` captures, and the
`SamsungOpenRing` app). Initial hypothesis: the app's gesture handling pinned the
ring's IMU on. Found a genuine bug — a **tug-of-war**: `RingConnection` re-enabled
gestures every time Samsung's app disabled them, plus an unbounded always-on service.

**Correction after web research:** Samsung rates the ring at ~7 days *including*
gesture use, so gestures are cheap. The widespread "1 week → 1 day" drain is usually a
**stuck-sensor hardware fault** (warranty-replaced), a firmware regression, or the
ring left **out of its charging case**. The app's contribution is real but small.
See [BATTERY.md](BATTERY.md).

## 2. App fix (SamsungOpenRing)

Adversarially reviewed and corrected the app:

- Removed the gesture tug-of-war (respect Samsung's power-saving disable).
- Replaced a broken 60 s sliding watchdog with an **absolute, persisted session cap**
  (default 8 h) that honours the real trigger window, survives reconnects/restarts,
  and renews only while a trigger is genuinely active.
- Added `CONNECTION_PRIORITY_LOW_POWER` and ring-battery telemetry logging.
- Applied the same cap to the Tasker plugin's service.

Files: `SamsungOpenRing/core/.../RingConnection.kt`, `OpenRing.kt`,
`app/.../GestureService.kt`, `app/.../triggers/TriggerManager.kt`,
`tasker-plugin/.../TaskerGestureService.kt`, `SamsungOpenRing/README.md`.
All builds verified (`assembleDebug` / `assembleRelease`).

## 3. The Claude Bridge (instrument)

Built a **debug-build-only** loopback HTTP control surface (`ClaudeBridgeService`,
`app/src/debug/`) so the ring can be driven over ADB. Full raw command access by the
user's choice; loopback-bind + debug-gate are the only boundaries. Driven via
`SamsungOpenRing/tools/ring` and a stateful Python REPL `tools/ring_shell.py`.
See [CLAUDE-BRIDGE.md](CLAUDE-BRIDGE.md).

## 4. Live validation

With the phone tethered, connected to the ring as a second GATT client and validated
the protocol against the device. Confirmed / corrected and folded into **RFC Appendix
B (B.1–B.11)**:

- Full GATT database; TX/RX roles were reversed in the spec (corrected).
- Gesture path end-to-end (9 real pinches detected by our app and Samsung's).
- Battery `0b 0b 42 06 00 05 NN …` (note `0x42` response header) — ring read at 63%.
- Standard Heart Rate Service `180D`/`2A37` (live 78 bpm) — new, beyond the RFC.
- Wearing state `0b 0b 34 37 00/01` (§7.5) — confirmed by removing/replacing the ring.
- Heartbeat exactly every 600 s; low-power connection priority accepted.
- Second-client capability matrix (what the ring services for a non-primary client).

## 5. Cracking the eedd5e73 "rolling value"

A read-only characteristic pair (`a12be31c` / `50f98bfd`) returned a 16-byte value
that **changed on every read** (each byte 0–99, high entropy, no temporal
correlation → a nonce, not a sensor). Static analysis trail:

1. Absent from ringplugin, paranmgr, Samsung Health, Galaxy Wearable (DEX + native).
2. **Found in SmartThings** (`com.samsung.android.oneconnect`), as plain Java.
3. Decompiled `classes6/8/9.dex` with jadx and reconstructed the algorithm.

Result: `eedd5e73` = `UUID_SERVICE_AUTHENTICATION`; the value is a nonce; the
response is the nonce-IV AES encryption of the constant `"smartthings"`. Then traced
the key derivation and the full mutual-auth + encrypted command channel.
See RFC Appendix B.8 / B.12 (the decompiled-code-evidence writeup is kept private,
outside this repo).

## 6. Teardown + battery strategy

Disconnected the bridge + shell cleanly (ring stopped draining at ~52%). Produced a
ranked battery-longevity plan ([BATTERY.md](BATTERY.md)).
